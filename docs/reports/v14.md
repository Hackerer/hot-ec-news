# v14 报告

日期：2026-03-14

## 本轮自问自答

问题：

- 既然 `status` 已经能看状态了，为什么还要再做 `doctor`？

回答：

- 因为 `status` 解决的是“现在是什么状态”，`doctor` 解决的是“现在能不能上线跑”。需求方真正需要的是在开启日推前明确知道哪些点会导致交付失败。

## 本轮目标

- 增加生产预检命令
- 在开启自动推送前发现 misconfigured 配置
- 用 fail / warn / pass 表达风险等级

## 交付内容

- 新增 `src/pipeline/doctor.ts`
- 新增 `doctor` CLI 命令
- 新增 push readiness 抽象复用到 `status` 与 `doctor`
- 预检覆盖：
  - 类目
  - 第一信源
  - 第二信源
  - 工作区路径
  - 自动推送准备度
  - 调度状态

## 实际测试结果

已执行：

- `npm run typecheck`
- `npm test`
- 新增 doctor 单元测试
- 干净目录下验证 fail / warn / ready 三种预检结果

结果：

- TypeScript 类型检查通过
- `vitest` 共 38 个测试全部通过
- 缺 webhook 的 auto push 会被判定为 `fail`

- SMTP 配齐后会被判定为 `pass`
