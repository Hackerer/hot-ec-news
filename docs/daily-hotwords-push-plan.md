# 每日电商热词推送设计规划

调研日期：2026-03-14

当前需求范围：

- 主方向：电商搜索热词
- 聚焦类目：服饰、鞋靴、首饰
- 数据策略：平台自身数据为主，第三方平台数据作为第二信源做聚合与校验
- 运行环境：macOS 与 Windows 都能运行

## 1. 结论先行

不建议把这件事做成单纯的 `skill`。原因很简单：`skill` 适合“被调用时执行”，不适合“每天固定时间主动推送”。如果目标是每天稳定推送，主体应该是一个可调度的软件服务，`skill` 只能作为辅助能力。

当前最可行的方案是：

1. 第一阶段做一个本地常驻小程序/脚本服务。
2. 核心程序用跨平台技术栈实现，分别接入 macOS 和 Windows 的系统调度器。
3. 用平台原始数据 + 第三方数据 + 浏览器自动化三种方式采集热词。
4. 用企业微信机器人、飞书机器人或邮件把日报推送给你。

我推荐的主方案是：

- 形态：本地常驻程序，而不是纯 skill。
- MVP 推送渠道：企业微信机器人或邮件。
- MVP 平台范围：
  - 电商主平台：淘宝/天猫、京东、拼多多、抖音商城
  - 内容辅助平台：抖音、小红书
  - 第三方第二信源：蝉妈妈、飞瓜数据、千瓜数据、魔镜洞察
- 类目优先级：服饰 > 鞋靴 > 首饰。

原因是你的目标已经不是“泛热点”，而是“电商选品/内容/投放”所需的类目热词；这类需求更依赖电商平台自身信号，再用第三方行业平台做补强和校验。

## 2. 深度调研结论

### 2.1 平台官方开放能力现状

截至 2026-03-14，我查到的官方/准官方资料显示：

- 抖音开放平台当前公开强调的是账号运营、视频/用户/经营数据、小程序、电商等能力。官方概述页提到平台提供“发布视频和获取视频数据、用户数据和各类榜单信息”等能力，但我检索到的公开文档里，没有看到一个明确可直接拿“搜索热词榜”的统一公开接口。
- 小红书开放平台当前公开文档主要覆盖订单、商品、库存、售后、素材、消息推送等商家/服务商能力。官方开放平台介绍页也明确写的是订单、商品、库存等接入接口，没有看到“热搜词/搜索热词”接口。
- 这意味着：跨平台“搜索热词”服务，不能假设存在一套稳定的官方 OpenAPI 可以直接打通。

结论：

- 官方能力可作为“补充数据源”，不能作为“唯一数据源”。
- 真正可落地的方案必须支持多种采集方式并存。

### 2.2 第三方平台可以用，但不能单独作为真值

可以，而且应该纳入设计。

我调研后认为，第三方平台最适合承担的是：

- 补全平台自身抓不到的类目趋势
- 对热词做交叉验证
- 提供达人、商品、类目维度的辅助解释
- 在官方入口波动时提供兜底信号

但不建议把第三方平台当成唯一数据源。原因是：

- 各家数据口径不同
- 更新时效不同
- 有些更偏内容热度，有些更偏商品交易
- 很多平台未公开标准化 API，往往需要商务采购、后台导出或定制接入

结论：

- 第三方平台适合做“第二信源”
- 平台自身数据仍然是第一优先级

### 2.3 “朋友圈热词”产品定义存在问题

这里有一个必须先纠正的产品定义问题。

“朋友圈”不是一个公开搜索型产品，严格说并不存在一个稳定可抓取的“朋友圈搜索热词榜”。如果继续沿用这个定义，系统会长期处于以下问题：

- 数据口径不稳定
- 合规风险高
- 技术实现依赖非官方自动化
- 推送结果不可验证

更合理的替代口径是：

- 微信搜一搜热词
- 微信指数
- 视频号/公众号搜索趋势（后续可选）

这里我明确标注：这是基于当前产品形态和公开能力做出的产品替代推断，不是来自单一官方页面的直接表述。

### 2.4 真正难点不在“定时”，而在“数据源”

定时推送本身非常容易，任何一种调度器都能做：

- macOS `launchd`
- Windows Task Scheduler
- GitHub Actions `schedule`
- Vercel Cron

真正难的是：

- 哪些平台能用官方接口
- 哪些平台只能抓公开网页
- 哪些平台要登录后才能看到热词
- 哪些平台只能在 App 里看到
- 哪些平台有反爬/风控
- 哪些平台的数据口径其实是“推荐热榜”而不是“搜索热词”

所以这件事本质上是“多平台情报采集系统”，不是“一个定时脚本”。

## 3. 方案选型

### 3.1 方案 A：做成 skill

优点：

- 接入当前工作流轻
- 可以复用模型做总结和去重

缺点：

- skill 无法自己每天主动唤起
- 仍然需要外部调度器
- 不适合管理登录态、浏览器会话、反爬重试

结论：

- 不适合作为主体方案
- 可以作为分析/摘要能力挂在系统后面

### 3.2 方案 B：纯云端服务

典型形态：

- GitHub Actions / Vercel Cron / 云函数
- 定时触发采集与推送

优点：

- 不依赖你的电脑开机
- 部署后维护方便
- 适合纯 HTTP 接口和公开网页

缺点：

- 对登录态、验证码、App 内数据、风控绕不过去
- GitHub Actions 定时触发会延迟；官方文档明确写了高负载时会延迟，最短间隔 5 分钟，公有仓库 60 天无活动还会自动停用定时工作流
- Vercel Cron 在 Hobby 计划下一天只能跑一次，且可能在指定小时内任意时间触发；时区固定是 UTC；官方还写了失败不自动重试

结论：

- 适合作为“公开网页/公开接口”的定时层
- 不适合作为全部数据源的唯一承载方案

### 3.3 方案 C：本地常驻程序

典型形态：

- 本地运行一个 Node.js 服务
- macOS 用 `launchd`，Windows 用 Task Scheduler
- Playwright 复用已登录浏览器会话

优点：

- 最容易保留登录态
- 最适合处理国内平台的反爬、Cookie、扫码登录
- 可以直接控制浏览器、截图、导出、调试
- 本地时区天然准确

缺点：

- 你的电脑要开机
- 稳定性取决于本机网络和浏览器环境

结论：

- 最适合当前需求的 MVP

### 3.4 方案 D：混合架构

典型形态：

- 云端做调度、存储、推送
- 本地或专用节点做复杂采集

优点：

- 稳定性更高
- 平台适配能力最强
- 后续可扩展成团队级产品

缺点：

- 系统复杂度最高
- 成本和维护成本上升

结论：

- 适合作为第二阶段目标

## 4. 推荐方案

### 4.1 最终推荐

我建议采用：

**跨平台本地常驻程序 + 可插拔采集器 + 第三方校验层 + 企业微信/邮件推送**

这是当前“可做、可维护、可验证”的最优平衡点。

### 4.2 为什么不直接上云

因为你要的不是单一网站监控，而是跨平台、跨类目、跨信源的电商热词。

在中国社交/电商平台里，这类数据通常分五种来源：

1. 官方商家/搜索数据
2. 官方公开网页榜单
3. 第三方行业数据平台
4. 登录后页面中的榜单/搜索建议
5. App 内部页面

第 3、4、5 类往往决定了数据质量上限，而它们通常不适合纯 serverless。

## 5. 产品方案

### 5.1 产品目标

每天固定时间，向你推送一份“服饰/鞋靴/首饰三类目”的电商热词日报，帮助你快速发现：

- 今天用户在搜什么
- 哪些词在多个电商平台同时升温
- 哪些词由内容平台扩散到交易平台
- 哪些词值得当天追内容、追选品、追投放

### 5.2 推送内容结构

建议日报结构如下：

1. 今日总览
2. 服饰热词
3. 鞋靴热词
4. 首饰热词
5. 跨平台共振词
6. 第三方校验结果
7. 新增爆发词
8. 连续上榜词
9. 备注与异常说明

每个词建议包含：

- 平台
- 排名或热度分
- 相比昨日变化
- 采集时间
- 原始入口链接或截图指针
- 自动标签
  - 内容向
  - 商品向
  - 节日向
  - 品牌词
  - 需求词

### 5.3 首版平台清单

电商主平台：

- 淘宝/天猫
- 京东
- 拼多多
- 抖音商城

内容辅助平台：

- 抖音
- 小红书

第三方第二信源：

- 蝉妈妈
- 飞瓜数据
- 千瓜数据
- 魔镜洞察

后续可选：

- 快手电商
- 得物
- 微信搜一搜 / 微信指数
- 美团闪购

### 5.4 热词评分模型

建议不要只转发“原榜单”，而是做一个统一评分：

统一热词分 = 平台内热度归一化分 + 平台权重 + 趋势变化分 + 跨平台共现分 + 信源可信度分

建议首版参数：

- 平台内热度归一化：50%
- 趋势变化：20%
- 跨平台共现：20%
- 信源可信度：10%

人工规则示例：

- 节日词加权
- 类目词加权
- 品牌词去噪
- 无商业价值娱乐词降权

## 6. 技术方案

### 6.1 系统架构

建议拆成 6 层：

1. 调度层
2. 采集层
3. 信源校验层
4. 规范化层
5. 分析层
6. 推送层

### 6.2 各层职责

#### 调度层

MVP：

- macOS：`launchd`
- Windows：Task Scheduler

备选：

- GitHub Actions
- Vercel Cron

#### 采集层

每个平台一个 adapter：

- `taobaoCollector`
- `jdCollector`
- `pddCollector`
- `douyinMallCollector`
- `douyinContentCollector`
- `xiaohongshuCollector`
- `thirdPartyChanmamaCollector`
- `thirdPartyFeiguaCollector`
- `thirdPartyQianGuaCollector`
- `thirdPartyMagicMirrorCollector`

每个 adapter 输出统一结构：

```json
{
  "platform": "douyin",
  "source_type": "official_page",
  "keyword": "防晒衣",
  "rank": 3,
  "score_raw": "892341",
  "captured_at": "2026-03-14T09:00:00+08:00",
  "entry_url": "https://example.com",
  "snapshot_path": "snapshots/douyin/2026-03-14/01.png"
}
```

#### 信源校验层

这一层专门做聚合和校验：

- 同关键词跨来源合并
- 平台信源与第三方信源比对
- 打可信度标签
- 生成“已验证 / 待验证 / 单点来源”状态

建议首版规则：

- 同一关键词出现在 1 个平台第一信源，可直接入榜
- 同一关键词只出现在第三方平台时，至少需要 2 个来源同时出现才入榜
- 同一关键词若第一信源与第二信源方向一致，则提高可信度

#### 规范化层

做以下处理：

- 同义词归并
- 品牌词识别
- 噪声词过滤
- 类目打标
- 热度归一化

#### 分析层

输出三类结果：

- 平台榜单
- 趋势变化
- 跨平台共振
- 第三方一致性校验

#### 推送层

首版建议支持：

- 企业微信机器人
- 邮件

第二阶段再加：

- 飞书机器人
- Telegram
- Notion/Google Sheets 归档

### 6.3 技术栈建议

我建议选 Node.js，而不是纯 Python。

原因：

- Playwright 在 Node 生态更成熟
- 浏览器自动化和页面选择器管理更方便
- 适合做后续 Web 控制台
- 推送、存储、调度集成成本低

推荐栈：

- Runtime: Node.js 22
- Browser automation: Playwright
- Storage: SQLite
- Scheduler adapters: macOS `launchd` + Windows Task Scheduler
- Push: 企业微信 webhook / SMTP
- Config: `.env` + `config/platforms.json`
- Report rendering: Markdown + HTML 模板

如果后续你更希望做数据分析而不是浏览器采集，可以增加：

- Python 分析脚本
- pandas
- DuckDB

但不建议首版就把主流程放在 Python 上。

跨平台原则：

- 核心逻辑只写一份
- 采集器不写操作系统分支
- 调度与开机自启用 adapter 抽象
- 浏览器会话和缓存分别落到系统标准用户目录

### 6.4 目录结构建议

```txt
hot_ec_news/
  docs/
  src/
    collectors/
    normalizers/
    analyzers/
    pushers/
    schedulers/
  data/
    raw/
    normalized/
    reports/
    snapshots/
  config/
    platforms.example.json
  scripts/
    run-daily.ts
    backfill.ts
```

## 7. 数据源策略

### 7.1 数据源优先级

按信源优先级从高到低：

1. 官方商家/搜索工具
2. 官方公开榜单页
3. 第三方数据平台
4. 登录后网页数据
5. App 自动化抓取

原则：

- 第一信源负责定主口径
- 第二信源负责校验和补全
- 自动化只解决官方入口缺失问题

### 7.2 每个平台建议

#### 抖音 / 抖音商城

优先顺序：

1. 官方开放能力中可直接获取的榜单/经营数据
2. 官方公开热榜页
3. 第三方数据平台
4. Playwright 登录态采集

备注：

- 公开文档显示抖音开放平台有榜单/经营相关能力，但是否能直接覆盖“搜索热词”要逐接口验证。
- 可引入蝉妈妈、飞瓜数据作为第二信源。

#### 小红书

优先顺序：

1. 官方公开页
2. 第三方数据平台
3. 登录态网页采集
4. 必要时 App 侧自动化

备注：

- 当前公开开放平台文档更偏商家履约接口，不应把“热搜词”寄希望于官方 OpenAPI。
- 可引入千瓜数据作为第二信源。

#### 淘宝/天猫、京东、拼多多

优先顺序：

1. 商家后台中的关键词/搜索词工具
2. 搜索建议词
3. 第三方行业平台
4. 官方或半官方分析产品可见榜单
5. 自动化采集

备注：

- 这几个平台首版不应承诺“百分之百官方 API 接入”，更现实的是“多源拼接”。
- 可引入魔镜洞察这类行业平台做品类趋势补充，但不替代平台一手口径。

## 8. 合规与风险评估

### 8.1 风险类型

主要风险有：

- 平台服务条款限制抓取
- 第三方平台授权与采购限制
- 登录态失效
- 验证码/风控
- 页面结构频繁改版
- 移动端专属入口无法稳定复用

### 8.2 风险控制策略

建议采取：

- 严格限频
- 每个平台单独可关闭
- 每个第三方信源单独配置开关和授权状态
- 保留快照，方便校验
- 采集失败不推空数据，改推“平台异常”
- 统一异常分级

### 8.3 产品侧风险声明

日报里建议明确写：

- 数据采集时间
- 数据来源类型
- 该词是否来自官方榜单
- 该平台当日是否存在缺失

这样可以避免用户把日报误解为“官方最终口径”。

## 9. MVP 实施路线

### 9.1 第一周

- 明确类目规则（服饰、鞋靴、首饰）
- 确定推送渠道
- 跑通 2 个电商平台采集
- 跑通 1 个第三方平台采集
- 跑通日报生成与推送

### 9.2 第二周

- 扩到 4 到 6 个平台/信源
- 建立热词规范化和去重
- 建立第三方校验分
- 加入历史对比

### 9.3 第三周

- 增加快照留存
- 增加失败重试和告警
- 增加配置化平台开关
- 增加 Windows 调度安装脚本

## 10. 我建议你现在就先定的三个产品决策

如果要进入实现，我建议先锁定这三件事：

1. 推送到哪里
2. 每天几点推
3. 第一批到底盯哪些平台和第三方信源

建议默认值：

- 推送渠道：企业微信机器人 + 邮件备份
- 推送时间：每天 09:00（Asia/Shanghai）
- 第一批平台：淘宝/天猫、京东、拼多多、抖音商城、抖音、小红书
- 第一批第三方信源：蝉妈妈、飞瓜数据、千瓜数据
