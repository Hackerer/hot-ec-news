## 1. 开发目标

开发一个可在 `macOS` 和 `Windows` 运行的本地程序，每天定时采集并推送电商搜索热词，聚焦以下 3 个类目：

- 服饰
- 鞋靴
- 首饰

系统需要满足：

- 平台一手数据为主
- 第三方平台数据作为第二信源做校验与补全
- 支持日报推送
- 支持保留快照与历史数据
- 支持后续扩展更多平台和类目

## 2. 交付分层

### 2.1 MVP

目标：先跑通“每天稳定收到一份可用日报”。

MVP 范围：

- 平台主信源：淘宝/天猫、京东
- 内容辅助信号：抖音
- 第三方第二信源：蝉妈妈或飞瓜数据，至少接 1 个
- 类目：服饰、鞋靴、首饰
- 推送渠道：企业微信机器人或邮件
- 系统支持：macOS、Windows

MVP 不做：

- 全量平台覆盖
- Web 控制台
- 多用户
- 复杂权限系统
- 云端部署

### 2.2 V1

目标：把系统从“可用”提升到“稳定、可扩展、可比较”。

V1 扩展范围：

- 增加拼多多、抖音商城、小红书
- 第三方信源扩展到 2 到 3 个
- 引入历史趋势、涨跌幅、连续上榜
- 增加 Windows 调度安装脚本和一键配置
- 增加异常告警和采集重试

## 3. 阶段计划

建议按 6 个阶段推进，总周期按单人开发估算为 `3 到 4 周`。

### 阶段 0：需求锁定与环境初始化

预计：0.5 到 1 天

目标：

- 锁定第一批平台
- 锁定第一批第三方信源
- 锁定推送渠道
- 初始化项目骨架

任务：

- 确认 MVP 平台顺序
- 确认是否有第三方平台账号或导出权限
- 确认企业微信机器人或 SMTP 凭据
- 初始化 Node.js 项目结构
- 建立 `.env.example`、`config/platforms.example.json`

交付物：

- 项目基础目录
- 环境变量模板
- 平台配置模板

验收标准：

- 本地可以启动基础 CLI
- macOS 和 Windows 都能执行同一个入口命令

### 阶段 1：核心框架与数据模型

预计：1.5 到 2 天

目标：

- 建立统一的数据流和存储结构

任务：

- 定义采集结果标准结构
- 定义标准化后热词结构
- 定义日报结构
- 建立 SQLite schema
- 搭建日志、配置、存储目录
- 建立 collector / validator / analyzer / pusher 模块边界

交付物：

- 数据 schema
- SQLite 初始化脚本
- 基础模块骨架

验收标准：

- 任意一个 mock collector 输出的数据可以被入库
- 可以从数据库读取并生成一份 mock 日报

### 阶段 2：第一信源采集器

预计：4 到 6 天

目标：

- 跑通 2 个主平台采集

任务：

- 实现 `taobaoCollector`
- 实现 `jdCollector`
- 实现 `douyinContentCollector` 或 `douyinMallCollector`
- 为每个采集器增加：
  - 配置项
  - 限频
  - 快照
  - 错误分类
- 建立原始数据落盘规则

交付物：

- 2 到 3 个可运行采集器
- 原始数据样本
- 快照样本

验收标准：

- 每个采集器能稳定产出 20 条以上热词
- 热词可被正确归入服饰、鞋靴、首饰或未分类

### 阶段 3：第二信源与校验层

预计：2 到 4 天

目标：

- 引入至少 1 个第三方平台，并建立可信度规则

任务：

- 接入 `蝉妈妈` 或 `飞瓜数据`
- 定义信源优先级与可信度分
- 实现同词跨信源合并
- 实现“已验证 / 待验证 / 单点来源”状态
- 建立词级别校验日志

交付物：

- 1 个第三方采集器
- 信源校验模块
- 可信度打分结果

验收标准：

- 同一关键词来自不同信源时可以归并
- 日报中能显示该词是否通过第二信源校验

### 阶段 4：类目识别、分析与日报生成

预计：2 到 3 天

目标：

- 让日报从“原始热词列表”升级为“可读的电商情报”

任务：

- 建立类目词典和规则
- 建立品牌词、噪声词过滤
- 实现热度归一化
- 实现涨跌幅计算
- 实现跨平台共振词检测
- 生成 Markdown 和 HTML 日报

交付物：

- 分析模块
- 日报渲染模板
- 示例日报

验收标准：

- 每天生成的日报能分出 3 个核心类目
- 可以看到新增词、共振词、连续词

### 阶段 5：推送与跨平台调度

预计：1.5 到 2.5 天

目标：

- 让程序每天自动跑起来并发出推送

任务：

- 实现企业微信 webhook 推送
- 实现 SMTP 邮件推送
- 实现统一命令入口：`run-daily`
- 编写 macOS `launchd` 安装脚本
- 编写 Windows Task Scheduler 安装脚本
- 增加失败重试和退出码

交付物：

- 推送模块
- 调度安装脚本
- 定时运行说明

验收标准：

- 在 macOS 和 Windows 上均可完成定时安装
- 到点后可自动收到日报

### 阶段 6：稳定性与可维护性

预计：2 到 3 天

目标：

- 降低后续维护成本

任务：

- 增加重试、超时、限频、熔断
- 增加平台开关和信源开关
- 增加失败截图与错误归档
- 增加历史数据清理策略
- 增加基础测试
- 补充运行文档

交付物：

- 稳定性增强
- 运维文档
- 测试与样本数据

验收标准：

- 单个平台失败不会导致整份日报失败
- 失败平台会在推送中明确标注

## 4. 工作流拆分

建议并行拆成 5 条工作流：

### 4.1 基础设施流

负责：

- 项目骨架
- 配置系统
- SQLite
- CLI
- 调度安装脚本

### 4.2 数据采集流

负责：

- 官方入口采集
- 登录态维护
- 快照保留
- 第三方平台接入

### 4.3 数据治理流

负责：

- 类目识别
- 同义词归并
- 品牌词与噪声词处理
- 信源可信度模型

### 4.4 日报与推送流

负责：

- 模板渲染
- 企业微信推送
- 邮件推送
- 错误通知

### 4.5 测试与验收流

负责：

- 采集器回归验证
- Windows/macOS 兼容性
- 定时执行验证
- 日报质量抽样检查

## 5. 任务优先级

优先级 `P0`：

- 基础项目骨架
- SQLite schema
- 2 个一手采集器
- 1 个第三方采集器
- 日报生成
- 企业微信或邮件推送
- macOS / Windows 调度

优先级 `P1`：

- 拼多多、抖音商城、小红书扩展
- 历史趋势
- 连续上榜
- 失败重试
- 快照管理

优先级 `P2`：

- Web 控制台
- 多用户
- 云端同步
- 可视化分析面板

## 6. 关键依赖

以下事项会直接影响开发节奏：

- 是否有第三方平台账号、试用权限或数据导出能力
- 是否能提供企业微信机器人 webhook 或邮箱配置
- 是否能提供至少一个平台的登录态验证环境
- 是否接受首版先以“部分平台 + 可用日报”为目标

## 7. 主要风险

### 7.1 数据源风险

- 官方页面改版
- 登录态过期
- 采集频率触发风控
- 第三方平台没有开放接口

应对：

- 采集器模块化
- 单平台开关
- 快照和错误归档
- 第二信源兜底

### 7.2 类目识别风险

- 热词可能跨类目
- 品牌词和泛词噪声高

应对：

- 先规则后模型
- 建立人工白名单和黑名单
- 持续修正词典

### 7.3 跨平台兼容风险

- macOS 与 Windows 的路径、调度、浏览器缓存目录不同

应对：

- 路径统一走平台适配层
- 调度脚本独立抽象
- 核心逻辑不写 OS 分支

## 8. 验收里程碑

### 里程碑 A：MVP 可运行

标准：

- 2 个平台采集成功
- 1 个第三方平台接入成功
- 日报可生成
- 可手动触发推送

### 里程碑 B：双端定时可用

标准：

- macOS 定时可用
- Windows 定时可用
- 推送稳定

### 里程碑 C：V1 稳定

标准：

- 4 到 6 个平台/信源
- 历史趋势和校验分可用
- 异常处理完备

## 9. 建议的实际执行顺序

不要一开始就追求“全平台全类目”。建议按下面顺序做：

1. 先完成项目骨架和统一数据结构。
2. 先打通淘宝/京东两条主信源。
3. 再接一个第三方信源做校验。
4. 再做日报生成和推送。
5. 最后再补 Windows 定时安装、更多平台和稳定性。

这个顺序的好处是，最短路径就能验证“这件事到底有没有业务价值”，不会过早陷入平台接入细节。

## 10. 下一步建议

如果直接进入开发，我建议按下面顺序马上开工：

1. 初始化 Node.js 项目骨架与目录结构。
2. 定义 SQLite schema 和通用热词数据结构。
3. 先实现 `taobaoCollector`、`jdCollector`、`wecomPusher`。
4. 跑通一次手动日报。
5. 再接入第一个第三方平台。
