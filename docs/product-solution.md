# 产品方案

更新时间：2026-03-14

## 1. 产品定位

`hot-ec-news` 不是一个简单的“榜单搬运脚本”，而是一个面向电商运营、选品和内容投放的 `类目热词雷达`。

聚焦对象：

- 类目：服饰、鞋靴、首饰
- 用户：电商运营、内容运营、投放、选品
- 输出：每天一份可被直接消费的热词情报

## 2. 借鉴的产品精髓

### 2.1 TrendRadar：多源聚合，不做单榜单转发

可借鉴点：

- 多平台聚合
- 关键词过滤
- 多渠道推送
- 更像“情报产品”而不是“采集工具”

映射到当前产品：

- 不只保留淘宝/京东原始词
- 要输出跨平台共振、第三方校验、异常说明
- 推送层要支持企业微信、邮件，并可继续扩展

参考：

- https://github.com/sansan0/TrendRadar

### 2.2 RSSHub：每个信源都是独立适配器

可借鉴点：

- 每个平台一个 route / adapter
- 新增数据源时不破坏主流程
- 统一输出结构

映射到当前产品：

- 淘宝、京东、第三方 CSV 导入都做成独立 adapter
- 后面接抖音商城、拼多多、小红书时继续按相同模式扩展

参考：

- https://github.com/DIYgod/RSSHub

### 2.3 Apprise：通知必须抽象成统一通道层

可借鉴点：

- 通知通道不能和业务流程耦死
- 推送渠道要统一抽象

映射到当前产品：

- `wecom`、`email` 不应直接散落在 CLI 或 pipeline 中
- 推送层要可配置、可 dry-run、可继续接飞书/Telegram

参考：

- https://github.com/caronc/apprise

### 2.4 Crawlee / Playwright：采集必须考虑可靠性

可借鉴点：

- 抓取流程要有重试、失败隔离、快照
- HTTP 和浏览器自动化要分层

映射到当前产品：

- 当前优先用 HTTP suggestions
- 后续登录态页面才上 Playwright
- 单平台失败不能拖垮整次日报

参考：

- https://github.com/apify/crawlee
- https://github.com/microsoft/playwright

## 3. 产品结构

每天输出一份日报，建议结构固定为：

1. 总览
2. 服饰热词
3. 鞋靴热词
4. 首饰热词
5. 第三方校验结果
6. 异常说明

每个词至少包括：

- 关键词
- 综合分
- 最优排名
- 来源平台
- 是否已被第二信源校验
- 置信度

## 4. 配置化产品能力

产品级配置项应该暴露给用户，而不是写死在代码里：

- 哪些信源启用
- 哪些第三方导入启用
- 哪些推送渠道启用
- 是否每天自动推送
- 每天几点调度
- 各类目种子词

这也是本轮代码改造的重点方向。

## 5. 当前产品方案结论

当前最合理的产品形态是：

`配置驱动的跨平台电商热词雷达`

它的基本原则是：

- 主信源决定主口径
- 第二信源做校验和补全
- 推送层独立
- 平台接入可插拔
- 报告对异常透明
